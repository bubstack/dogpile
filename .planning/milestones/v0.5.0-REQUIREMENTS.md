## v1 Requirements

### Provenance Annotations

- [x] **PROV-01**: Caller can read provenance metadata (model id, provider id, call id, ISO-8601 start/end timestamps) on model-request and model-response events in a completed trace — **VALIDATED Phase 6**
- [x] **PROV-02**: Provenance fields are JSON-serializable and survive a round-trip through `replay()` — **VALIDATED Phase 6**

### Structured Event Introspection

- [x] **INTR-01**: Caller can filter completed trace events by type, agent id, turn number, or cost range using a typed query function — **VALIDATED Phase 7**
- [x] **INTR-02**: Introspection query returns typed subsets of `RunEvent[]` with no new types introduced (callers narrow, not cast) — **VALIDATED Phase 7**

### Health Diagnostics

- [x] **HLTH-01**: Caller can read a structured health summary on `RunResult` with machine-readable anomaly codes (`runaway-turns`, `budget-near-miss`, `empty-contribution`, `provider-error-recovered`) and configurable thresholds — **VALIDATED Phase 7**
- [x] **HLTH-02**: Health summary is computed at result time, available on `replay()`, and re-computed identically from the same trace on any runtime — **VALIDATED Phase 7**

### Audit Event Schema

- [x] **AUDT-01**: Caller can produce a versioned (`auditSchemaVersion: "1"`) audit record from a completed trace using a pure function — not auto-attached to `RunResult` — **VALIDATED Phase 8**
- [x] **AUDT-02**: `AuditRecord` type is independent of `RunEvent` schema and is validated by a frozen JSON fixture test that must be explicitly updated when the schema changes — **VALIDATED Phase 8**

### OTEL Tracing Bridge

- [x] **OTEL-01**: Caller can inject a duck-typed OTEL-compatible tracer on `EngineOptions` (no `@opentelemetry/*` import required) and receive spans for run start/end, sub-run start/end, and agent turn start/end — **VALIDATED Phase 9**
- [x] **OTEL-02**: Sub-run spans are nested under parent run spans reflecting the `parentRunIds` ancestry chain — delegate child runs do not appear as disconnected root traces in OTEL backends — **VALIDATED Phase 9**
- [x] **OTEL-03**: Tracer injection is optional; runs complete with no span overhead and no observable behavior change when no tracer is provided — **VALIDATED Phase 9**

### Metrics / Counters

- [x] **METR-01**: Caller can supply a metrics hook on `EngineOptions` and receive named counters (token usage, cost, turn count, duration) at run completion and sub-run completion — **VALIDATED Phase 10**
- [x] **METR-02**: Metrics hook is optional; runs complete with no overhead when no hook is provided — **VALIDATED Phase 10**

## Future Requirements (Carried to v0.6.0+)

- Per-turn health streaming (health diagnostics emitted as events during a run, not only at completion)
- Caller-defined-tree API: `Dogpile.nest({ children: [...] })` (from v0.4.0 deferred list)
- Cross-protocol shared transcript across parent/child boundary
- Per-child retry policy on `delegate` decisions
- `compactProvenance` deduplication mode (if trace size becomes a measured problem)
- Built-in OTLP HTTP exporter (caller owns exporters; defer indefinitely)
- `DogpileSpan.end()` timestamp parameter for sub-millisecond span accuracy

## Out of Scope (Confirmed)

- **`@opentelemetry/*` as runtime or peer dependency** — pure-TS runtime constraint satisfied
- **File/disk audit log writer** — caller owns persistence
- **PII redaction** — caller policy
- **Bundled metric exporters** (Datadog, StatsD, OTLP HTTP)
- **`perf_hooks` or `process.hrtime`** — timestamps use `Date.now()` / Web Performance API
- **Singleton global tracer registration** — tracer always caller-supplied per engine instance
- **Provenance on `model-output-chunk` events** — hot-path exclusion confirmed
