▼ Audit Function API 0 / 3
Q-01 Function input type
The audit function must accept a completed run to produce AuditRecord. RunResult is the standard caller-facing return value. Trace is the canonical JSON-serializable replay artifact — important for audit generation from stored traces (e.g., from a DB) where the full RunResult may not be available. Phase 7's computeHealth() took a Trace for the same reason.

▼ AuditRecord Field Set 0 / 4
Q-04 Fields beyond the required minimum
AUDT-01 success criteria specifies: auditSchemaVersion: '1', runId, intent, timestamps (startedAt/completedAt), agentCount, outcome, cost. RunMetadata already collects protocol, tier, modelProviderId — all derivable from Trace fields. AuditRecord must stay independent of RunEvent schema (AUDT-02), so any field must come from Trace-level data, not from parsing individual RunEvent variants.

Q-05 "outcome" field values
Outcome must be derivable from Trace.events without importing RunEvent types into AuditRecord. FinalEvent (type='final') = run completed successfully. BudgetStopEvent (type='budget-stop') = stopped by cap. AbortedEvent is a streaming-only event, not stored in Trace.events. For stored traces, abort is inferred by absence of 'final' or 'budget-stop'. The termination field on FinalEvent can provide additional detail.

Q-06 Cost representation in AuditRecord
AuditRecord must be independent of RunEvent schema — but CostSummary (usd: number) is a stable, simple type that isn't an event type. RunUsage (src/types.ts ~1461) adds inputTokens/outputTokens. Both are derivable from Trace.finalOutput or the FinalEvent. The question is how much token detail belongs in an audit record.

Q-07 terminationReason field
When outcome is 'budget-stopped', BudgetStopEvent.reason carries the specific cap class (BudgetStopReason). For compliance, knowing which cap triggered the stop may be material. This would be optional (present only when outcome !== 'completed').

▼ Agent and Sub-Run Handling 0 / 3
Q-08 Agent-level detail
AUDT-01 requires agentCount. Trace.agentsUsed is a readonly AgentSpec[] with id and role. TurnEvent/BroadcastEvent carry per-agent cost. For compliance, per-agent cost breakdown is sometimes required. STATE.md decision: 'AuditRecord is not derived from RunEvent via Pick/Omit' — any per-agent data must be its own AuditRecord-internal type.

Q-09 Sub-run (delegate child) representation
When a coordinator delegates child runs, Trace.events contains SubRunCompletedEvent/SubRunFailedEvent entries. Each carries a childRunId and an embedded child Trace. Phase 8 depends on Phase 7. STATE.md notes Phase 6 (event-shape) must complete before OTEL (Phase 9) — audit depends on stable provenance. The question is whether audit records are recursive or flat.
Chat more
Q-10 agentCount semantics
Trace.agentsUsed contains the agents configured for the run. In a broadcast run this is all agents. In a coordinator run, the coordinator plus its delegates are in agentsUsed. Sub-run agents are in child traces, not parent agentsUsed.
